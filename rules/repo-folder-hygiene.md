# Repo Folder Hygiene

Humans navigate a folder best when it holds **‚Č§10 items**. Architect every directory around that ceiling. A repo that dumps loose screenshots, scratch `.md`, brainstorm catalogs, and build artifacts into top-level (or any) folders is cluttered ‚ÄĒ clutter is drift, fixed in-turn per `drift-detection`. Keep only necessary files; everything generated is gitignored, never tracked.

## The ‚Č§10 rule

- **A folder should hold ‚Č§10 direct items** (files + subfolders). Past ~10, a human can't scan it at a glance ‚ÄĒ split into subfolders by concern, or absorb single-callers into their owner per `context-spillover`.
- **Angular's colocation is the gold standard**: a component folder = `x.component.ts` + `x.component.html` + `x.component.css` (+ `x.component.spec.ts`, `x.service.ts`, an `index.ts`, a `README.md` ‚ÄĒ still ‚Č§10). One feature, one folder, a handful of tightly-related files. Mirror this shape everywhere: a worker feature module (`manifest.ts` + `schemas.ts` + `service.ts` + `handlers.ts` + `README.md` + `__tests__/`) is the same idea per `feature-module-architecture`.
- **Top-level repo root is sacred** ‚ÄĒ it holds project meta only (README, CLAUDE.md, package.json, config, ONE canonical of each doc class). NO loose screenshots, NO `_ideas-*.md`/`_audit.md` scratch, NO `qa-shots/`, NO debug PNGs. Those go in their proper home or get deleted.

## Keep only necessary files

- **One canonical per doc class** ‚ÄĒ one `PROGRESS.md`, one checklist, one `ARCHITECTURE.md`, one ideas/backlog. Collapse `_ideas-100-v2/v3/allstar/refreshed` brainstorm duplicates into ONE archive (`docs/IDEAS.md`); a doc that only restates another becomes a pointer or is deleted.
- **Absorb single-callers** ‚ÄĒ a one-export helper used by exactly one module folds into that module. Two files defining the same shape collapse to one (per `zod-everywhere`).
- **Delete dead surface** ‚ÄĒ unmounted components, dead flags, unreachable routes, abandoned scripts, superseded docs. Grep importers/callers FIRST, then delete; every removal is reversible via git history.
- **Fold good ideas, drop the rest** ‚ÄĒ a scratch/idea file with something valuable: fold the idea into the real owner (code or canonical doc), then delete the scratch. Obsolete ideas are just deleted.

## `.gitignore` is the first line of defense (maintain it per project)

Every generated/regenerated output is gitignored + untracked ‚ÄĒ NEVER committed. On any project, keep `.gitignore` current with the best-appropriate set for its stack:

- **Always**: `node_modules/`, `dist/`, `build/`, `.cache/`, `coverage/`, `*.log`, `*.tsbuildinfo`, `.env*` (never commit secrets), `.DS_Store`, `*.tmp`, `*-backup-*`.
- **Test/visual artifacts**: ad-hoc screenshot dirs (`**/__visual-qa__/`, `e2e/screenshots/`, `qa-shots/`), Playwright `test-results/`, `playwright-report/`, `blob-report/`. KEEP Playwright `*-snapshots/` baselines + persistent state (`__seen-routes__.json`) tracked ‚ÄĒ those are source-of-truth, not artifacts.
- **Framework**: Angular `.angular/`; Vite `.vite/`; Next `.next/`; Cloudflare `.wrangler/`, `.dev.vars`; Python `__pycache__/`, `.ruff_cache/`, `.mypy_cache/`, `.venv/`.
- **Real assets STAY tracked** ‚ÄĒ `public/` brand/OG/icons, content images (`assets/blog-art`), favicons. The test is "regenerated by a build/test command?" ‚Üí ignore. "Authored once, a source-of-truth?" ‚Üí track.
- **NEVER blanket-ignore a content-asset extension** (`*.jpeg`, `*.png`, `*.webp`, `*.pdf`, `*.mp4`). It silently un-tracks real source assets ‚ÄĒ they still ship from local disk + build output, so nothing breaks for YOU, but a clone/CI/rebuild is missing them. **macOS (case-insensitive) hides it; a fresh Linux CI checkout exposes it** ‚ÄĒ a key reason CI runs the build on a clean tree, not just locally. Ignore SPECIFIC scratch paths (`tmp/*.jpeg`, `_scratch/`), never a bare `*.ext`. Reference incident (nyfoldingbox.com, 2026-06-20): a blanket `*.jpeg` excluded a featured image from git ‚ÄĒ only its AVIF/WebP variants were tracked; the first CI run caught the untracked source. A `check-image-variants`-style gate (every referenced asset has a committed source) catches this at build.
- Use a maintained baseline (github/gitignore + `npx gitignore <stack>`) as the floor, then add project-specifics. Re-audit `.gitignore` whenever a new tool/artifact class appears (per `prompt-as-training-signal`).

## Audit cadence (when touching a repo)

- `git ls-files '*.png' | sed -E 's#/[^/]+$##' | sort | uniq -c | sort -rn` ‚ÄĒ any artifact dir with many tracked images that a test regenerates = untrack + ignore.
- `find . -maxdepth 1 -type f \( -name '*.png' -o -name '_*.md' \)` ‚ÄĒ loose root scratch = relocate or delete.
- `ls` any folder; >10 items ‚Üí split or absorb.
- After a hygiene sweep, run the project's verify gate so a wrong delete fails the gate, not prod.
